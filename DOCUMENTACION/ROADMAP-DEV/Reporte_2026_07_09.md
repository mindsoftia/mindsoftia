> **Propósito:** Documentar las optimizaciones implementadas en el módulo de Inventario (procesamiento de imágenes) y la reestructuración del Dashboard Principal (SaaS) hacia un enfoque estrictamente contable y financiero.

### 1. ROL
**Technical Writer Senior & Arquitecto Frontend.**

### 2. CONTEXTO Y ALCANCE
El sistema presentaba dos cuellos de botella significativos:
1. **Rendimiento en la subida de imágenes:** El peso de las fotografías originales subidas al catálogo de productos bloqueaba el DOM y generaba tiempos muertos (congelación de UI) durante la creación/edición.
2. **Dashboard Ineficiente:** El panel principal ("Centro de Control SaaS") contenía variables "hardcodeadas" y métricas de módulos no activos (Academias, Nómina), lo cual generaba ruido visual y nulo valor operativo para las empresas (tenants).

**Alcance de este documento:**
- Explicar la lógica de compresión en cliente implementada en el módulo de inventario.
- Detallar el rediseño de métricas y la inyección segura (Multi-Tenant) en el Dashboard.

### 3. AUDIENCIA OBJETIVO
Desarrolladores Frontend, Arquitectos Backend y Product Owners que requieran comprender la estructura actual del flujo de subida de archivos y el renderizado del dashboard contable.

### 4. ESTRUCTURA DEL DOCUMENTO

#### Resumen Ejecutivo
Se implementó un algoritmo de compresión de imágenes nativo en el navegador usando HTML5 Canvas antes de enviar el payload a Supabase, reduciendo el peso de los archivos drásticamente. Además, se limpió el Dashboard principal, conectándolo con las tablas reales de `pos_ventas`, `crm_terceros` y `accounts`, asegurando un entorno "Multi-Tenant".

#### Arquitectura / Lógica de Procesamiento

1. **Compresión de Imágenes (Browser-Side):**
   - **Antes:** Archivo original (ej. 5MB) -> Request a Supabase (Lento).
   - **Ahora:** Archivo original -> HTML Image -> Canvas Resize (Máx. 800x800) -> Blob JPEG (80% calidad, ~100KB) -> Supabase.
   - **Beneficio:** 90% de ahorro en ancho de banda, sin bloqueo del DOM.

2. **Gestión de Storage (Fire & Forget):**
   - El borrado de la imagen anterior durante una edición ya no pausa la ejecución de la UI. Se dispara de forma asíncrona (`storage.remove`), permitiendo que el modal se cierre al instante.

#### Requisitos Previos (Para mantener la arquitectura)
- Mantener activo el `useAuthStore` con el `tenantId` correcto, ya que el backend lo requiere para filtrar los KPIs (RLS y Scopes).
- No reactivar la carga directa de archivos `File` sin pasar por la función compresora, para evitar la saturación del bucket `archivos_empresas`.

#### Guía de Estructura de BD (Dashboard Contable)
El Dashboard ya no consulta tablas SaaS globales. Ahora consulta:
- `pos_ventas`: Para calcular los ingresos (`total_factura`) del día actual filtrando por `empresa_id`.
- `crm_terceros`: Conteo de clientes (terceros con `es_cliente = true` y `estado = true`).
- `accounts` (PUC): Conteo de las cuentas contables activas.

> [!WARNING]
> Si la estructura de la tabla `pos_ventas` se altera en futuras migraciones (ej. cambio de nombre a la columna `total_factura`), el endpoint del Dashboard (`DashboardController.php`) arrojará un error 500 y devolverá nulo al Frontend.
